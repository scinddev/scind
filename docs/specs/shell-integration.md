# Scind Shell Integration Specification

**Version**: 0.1.2-draft
**Date**: December 2024
**Status**: Design Phase

This document defines Scind's shell integration, including standard CLI completion and Docker Compose passthrough with delegated completion.

---

## Overview

Scind provides two complementary interfaces:

1. **`scind` CLI**: The primary command for managing workspaces, applications, ports, and configuration. Uses standard completion generated by Cobra.

2. **`scind-compose` shell function**: A context-aware passthrough to `docker compose` that automatically injects the correct project name and compose files based on the current directory or explicit flags. Uses delegated completion from Docker's own completion scripts.

This separation exists because:

- Docker Compose has ~20 subcommands with hundreds of flags and options
- Reimplementing completion for all of them would be significant work with no added value
- Users already know `docker compose` — we just make it workspace-aware
- Shell function + completion delegation achieves full docker compose completion "for free"

---

## CLI Completion

Scind provides shell completion for all commands.

### Installation

**Bash**:
```bash
source <(scind completion bash)
# Or add to ~/.bashrc:
# echo 'source <(scind completion bash)' >> ~/.bashrc
```

**Zsh**:
```bash
source <(scind completion zsh)
# Or add to ~/.zshrc
```

**Fish**:
```bash
scind completion fish | source
# Or save to completions directory
```

### Features

Completion supports:
- Command and subcommand names
- Flag names and values
- Workspace names (from registry)
- Application names (from current workspace)

## The `scind compose-prefix` Command

This command is the bridge between the shell function and Scind's context resolution. It outputs a docker compose command prefix that includes all necessary flags.

### Behavior

```bash
$ cd ~/workspaces/dev/frontend
$ scind compose-prefix
docker compose -p dev-frontend -f '/home/user/workspaces/dev/frontend/docker-compose.yaml' -f '/home/user/workspaces/dev/.generated/frontend.override.yaml'

$ cd ~/workspaces/dev
$ scind compose-prefix -a backend
docker compose -p dev-backend -f '/home/user/workspaces/dev/backend/docker-compose.yaml' -f '/home/user/workspaces/dev/backend/docker-compose.worker.yaml' -f '/home/user/workspaces/dev/.generated/backend.override.yaml'
```

### Flags

| Flag | Short | Description |
|------|-------|-------------|
| `--workspace` | `-w` | Specify workspace (overrides context detection) |
| `--app` | `-a` | Specify application (overrides context detection) |

> **Note**: There is no `--flavor` flag. Flavor changes require regeneration and can impact running applications, so flavor must be set explicitly via `scind flavor set` before using `scind-compose`.

### Output Format

- All paths are single-quoted to handle spaces and special characters
- Compose files are included in the correct order (base files first, then generated override)
- Project name follows the `{workspace}-{app}` convention
- Output is a single line suitable for `eval`

### Error Behavior

If context cannot be resolved, the command exits with code 5 and prints an error to stderr:

```bash
$ cd ~
$ scind compose-prefix
Error: No application context detected.

Either:
  1. Run from within an application directory
  2. Specify explicitly: scind-compose -a APP [command]

Available workspaces: scind workspace list
$ echo $?
5
```

### Exit Codes

`scind compose-prefix` uses standard Scind exit codes:

| Code | Meaning |
|------|---------|
| 0 | Success - prefix output to stdout |
| 1 | General error (configuration issues, file not found, etc.) |
| 5 | Context detection failed (no workspace/app found) |

The shell function `scind-compose` checks for empty output to detect failures and re-runs the command to display the error message. When `compose-prefix` succeeds, `scind-compose` returns the exit code from the underlying `docker compose` command.

## Usage Examples

### Basic Usage

```bash
# From within an application directory (context detected)
$ cd ~/workspaces/dev/frontend
$ scind-compose exec php bash
$ scind-compose logs -f
$ scind-compose up -d

# From workspace root with explicit app
$ cd ~/workspaces/dev
$ scind-compose -a backend exec php bash
$ scind-compose -a frontend logs -f php

# From anywhere with explicit workspace and app
$ scind-compose -w dev -a frontend ps
```

### Tab Completion Examples

```bash
# Complete subcommands
$ scind-compose [TAB]
build   config  create  down    events  exec    images  kill    logs ...

# Complete service names (from the resolved compose file)
$ scind-compose exec [TAB]
php     nginx   mysql   redis

# Complete docker compose flags
$ scind-compose exec -[TAB]
-d          --detach        -e          --env
--index     --no-tty        --privileged ...

# Complete with explicit app (scind flags first)
$ scind-compose -a [TAB]
frontend     backend     shared-db

$ scind-compose -a backend exec [TAB]
php     nginx   postgres    worker
```

## Known Limitations

### All Shells

1. **Docker completion must be installed**: If docker's shell completion isn't available, the delegation falls back to basic subcommand completion only.

2. **First invocation latency**: The first tab completion calls `scind compose-prefix`, which involves context detection. Subsequent completions in the same directory use shell caching.

3. **Path spaces**: Paths with spaces are supported but require careful quoting in `scind compose-prefix` output.

### Bash-Specific

1. **Requires bash-completion package**: The `_init_completion` helper function comes from bash-completion. Without it, completion may not work.

2. **COMP_LINE reconstruction**: Some docker completions use `COMP_LINE` for substring matching. Our reconstruction from words may lose original spacing.

### Zsh-Specific

1. **1-indexed arrays**: The completion function must account for zsh's 1-indexed arrays when calculating positions.

2. **Completion function detection**: Docker's completion might be `_docker`, `_docker-compose`, or loaded lazily. The script tries multiple approaches.

### Fish-Specific

1. **`complete -C` overhead**: Fish's completion delegation spawns a subshell, adding latency compared to bash/zsh.

2. **Completion caching**: Fish aggressively caches completions. If compose files change, completions may be stale until the shell restarts.

## Future Considerations

### Potential Enhancements

1. **Completion caching**: Cache the compose-prefix output for a short duration to reduce latency on repeated completions.

2. **Service name completion from YAML**: Instead of delegating to docker for service names, parse the compose file directly. This would work even when containers aren't running.

3. **Flavor-aware completion**: Show available flavors when completing after `-f`/`--flavor` flag.

4. **Custom shortcuts**: Allow users to define additional shortcuts in Scind config:
   ```yaml
   # ~/.config/scind/config.yaml
   shortcuts:
     artisan: "exec php php artisan"
     composer: "exec php composer"
   ```

   Enabling: `scind-compose artisan migrate` → `docker compose ... exec php php artisan migrate`

### Alternative Approaches Considered

1. **Integrated `scind compose` subcommand**: Would require reimplementing all docker compose argument parsing and completion. Rejected due to complexity and maintenance burden.

2. **Alias-only approach** (`alias dc='docker compose -p ... -f ...'`): Doesn't support dynamic context detection. Rejected.

3. **Separate commands per docker compose subcommand** (`scind-exec`, `scind-logs`, etc.): Fragments the interface unnecessarily. Rejected.
